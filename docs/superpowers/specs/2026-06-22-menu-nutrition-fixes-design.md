# Дизайн: 2 фікси нутрієнтів меню

**Дата:** 2026-06-22
**Статус:** затверджено до імплементації

## Контекст і проблеми

Дві пов'язані задачі в домені підрахунку нутрієнтів меню:

1. **«Брудні» ваги інгредієнтів.** Механізм доведення денного меню до цільової
   калорійності масштабує страви множником `k` і округлює ваги через
   `Math.round(quantity * k)` (мін. 5). Звідси числа на кшталт 83 г, 147 г, і
   сума → `servingSize` 342 г. Потрібно округлювати ваги інгредієнтів до 10, щоб
   ваги страв були «круглими».

2. **Неточне БЖВ власної (кастомної) страви.** `parseCustomFood` просить LLM
   напряму вгадати `per100` готової страви — слабка сторона моделі. Меню ж
   рахує точно через детерміновану таблицю `FOOD_TABLE` + `computeMealNutrition`,
   сумуючи інгредієнти.

## Зачеплені файли (поточний стан)

- `src/lib/menu/generateMenuWithAI.ts` — `scaleDayToTarget`, `scaleMealToCalories`,
  `adjustDeficientDays`, `mapDays`. Масштабування множить калорії/БЖВ на `k`
  напряму; при `k≈1` (±3%) виходить рано, лишаючи «брудні» LLM-ваги.
- `src/lib/menu/foodNutrition.ts` — `FOOD_TABLE`, `lookupFood`, `toGrams`,
  `computeMealNutrition` (детермінований підрахунок per-100g raw).
- `src/lib/menu/parseCustomFood.ts` — LLM `gpt-4o-mini`, промт просить `per100`
  напряму.
- `src/app/api/menu/food/parse/route.ts` — POST `{text, grams}` → `{parsed}`.
- `src/components/menuPage/AddCustomFoodSheet.tsx` — крок 1 (назва+вага),
  крок 2 (степер ваги + редаговані БЖВ через `per100`).
- `src/types/meals.ts` — `MealIngredient`, `CustomEntry`, `NutritionPer100`.

## Прийняті рішення (узгоджено з користувачем)

- **Округлення:** поріг — великі інгредієнти до 10, дрібні точно; з перерахунком.
- **БЖВ кастом-страви:** гібрид (розклад на інгредієнти + відкат на per100).
- **UX кастом-страви:** показати редагований розклад на інгредієнти.

## Секція A — Округлення ваг інгредієнтів

### Новий helper `roundQuantity(quantity, unit)`

У `generateMenuWithAI.ts` (або поряд із масштабуванням):

- `unit` ∈ {`г`,`гр`,`мл`} і `quantity ≥ 20` → округлити до найближчих **10**
  (мінімум 10);
- `unit` ∈ {`г`,`гр`,`мл`} і `quantity < 20` → лишити як є, мінімум **5**
  (спеції, олія, дріжджі не спотворюються);
- `unit` = `шт` → ціле число, мінімум **1** (як зараз).

### Зміна логіки масштабування

Замість прямого множення калорій/БЖВ на `k`:

1. Множимо **ваги інгредієнтів** на `k`.
2. Застосовуємо `roundQuantity` до кожного інгредієнта.
3. **Перераховуємо** калорії/БЖВ страви через `computeMealNutrition` — щоб
   цифри точно відповідали округленим вагам (макро↔вага консистентні).
4. `servingSize` округлюємо до найближчих 10.

### Окремий прохід у `mapDays`

Округлення+перерахунок — **окремий прохід, що виконується завжди**, навіть коли
`k` у межах ±3% (зараз рання виходка лишає брудні LLM-ваги). Порядок проходів:

1. normalize (parse + перший `computeMealNutrition`);
2. `scaleDayToTarget` (множник `k`, клемп `[0.5, 2.0]`);
3. `adjustDeficientDays` (borrow + re-scale);
4. **round + recompute** (новий прохід; зачіпає й borrowed-страви);
5. фінальні агрегати (`totalCalories`, `totalPrepMinutes`).

`scaleMealToCalories` (шлях свопів) отримує той самий фінальний round+recompute,
теж безумовно (прибрати ранній `return` для ваг — округлення застосовується
завжди, навіть якщо `k≈1`).

### Компроміс

Округлення дає дрейф ±кілька ккал на страву; денний тотал може злегка вийти за
±3% (на практиці < 5%). Прийнятна ціна за чисті ваги. Клемп `k` ∈ `[0.5, 2.0]`
і логіка borrow зберігаються.

### Тести

- `roundQuantity`: 23г→20, 147г→150, 15мл→15 (нижче порога), 2г→2, 3шт→3, 0→мін.
- Інтеграційно: після `mapDays` усі вагові інгредієнти ≥20 кратні 10;
  `servingSize` кратний 10; калорії страви == `computeMealNutrition(ingredients)`.

## Секція B — Бекенд кастом-страви: гібрид

### `parseCustomFood.ts` — новий промт

LLM **розкладає** страву на інгредієнти з вагами (схема як у меню:
`{name, quantity, unit, shoppingCategory}`) для заданої загальної ваги, **і**
додатково дає власну оцінку `per100` як запасний варіант. Один виклик OpenAI.

### Гібрид-рішення в коді

1. Рахуємо БЖВ через `computeMealNutrition` по інгредієнтах.
2. Рахуємо **покриття** = (грами інгредієнтів, що знайшлись у `FOOD_TABLE`) /
   (загальні грами вагових інгредієнтів). Потрібен доступ до результату матчингу —
   `lookupFood` повертає `null` для незнайдених; підсумувати `toGrams` лише по
   тих, де `lookupFood != null` (зеро-макро спеції рахуються як «знайдені»).
3. Якщо покриття ≥ **0.6** і калорії > 0 → беремо детермінований результат
   (`method: 'ingredients'`).
4. Інакше → відкат на LLM-`per100` (`method: 'estimate'`, поточна поведінка).
5. `per100` виводимо з обчисленого: `макро / сумаГрам × 100` — щоб степер ваги
   працював у режимі оцінки.

Поріг покриття 0.6 — тюнінговий; винести константою.

### Новий ендпоінт `POST /api/menu/food/compute`

- Приймає `{ ingredients: MealIngredient[], grams?: number }`.
- Повертає `{ calories, protein, fat, carbs, per100 }` через `computeMealNutrition`.
- **Без OpenAI** — чистий детермінований підрахунок. UI перераховує при правках
  інгредієнтів без AI-витрат.
- Gate: валідна сесія (`readSessionUserId`), **не** subscription (немає вартості).
- Валідація: масив непорожній, розумні межі quantity, обрізати довжину.

### Розширення відповіді `/parse`

`parsed` отримує поля:
- `ingredients: MealIngredient[]`
- `method: 'ingredients' | 'estimate'`

(зберігаючи наявні `name, emoji, calories, protein, fat, carbs, grams, per100, error`).

### Тести

- `parseCustomFood`: мок LLM з відомими інгредієнтами (висока покриваність) →
  `method='ingredients'`, БЖВ із таблиці; мок з екзотичними (низька покриваність)
  → `method='estimate'`, per100 від LLM.
- `compute`-ендпоінт: детермінований результат на фіксованому наборі інгредієнтів.

## Секція C — Фронтенд кастом-страви: редагований розклад

### `AddCustomFoodSheet.tsx`, крок 2 — два режими

**Режим інгредієнтів** (`method='ingredients'`):
- Список інгредієнтів із вагами — редагування ваги/назви, видалення рядка,
  додавання рядка.
- При зміні — дебаунс-виклик `POST /api/menu/food/compute` → оновлює БЖВ і `per100`.
- Загальна вага = сума інгредієнтів (показуємо, окремий степер ваги ховаємо).
- Підсумкові калорії/БЖВ — зверху, read-only (з підрахунку).

**Режим оцінки/ручний** (`method='estimate'` або «ввести вручну»):
- Поточна поведінка незмінна — степер ваги + редаговані БЖВ через `per100`.

### Тип `CustomEntry` (`meals.ts`)

Додати опційне `ingredients?: MealIngredient[]` — збережена страва зберігає
розклад (для майбутнього ре-едиту). Картка (`CustomEntryCard`) показує лише
тотали, як зараз. Зберігання inline у `weekly_menus` (як і решта `customEntries`).

### Тести

- Рендер режиму інгредієнтів: правка ваги рядка → виклик compute → оновлення БЖВ.
- Видалення/додавання рядка коректно тригерить перерахунок.
- Fallback-режим (estimate) поводиться як раніше.
