# Генерація меню — журнал експериментів зі швидкістю (R&D)

> **Статус**: експерименти проведені, у код НЕ влиті (відкладено). Застосовано лише
> зміну моделі `gpt-4o → gpt-4.1-mini`. Повна реалізація лежить у git stash —
> див. розділ «Як відновити».
> **Дата**: 2026-06-01

## Проблема

Користувач скаржився, що меню довго генерується. Початкова гіпотеза — «список
покупок формується надто довго (5-7с)». **Гіпотеза спростована вимірюванням.**

Додано інструментацію в `POST /api/menu/generate` (`console.time`-стиль навколо
кожної фази) і пер-спробний лог у генерацію. Результат першого заміру (gpt-4o, 7 днів):

```
[menu/generate timing] {"preWork":203,"aiGeneration":47051,"insertMenu":61,
  "buildShoppingList":0.2,"writeShoppingList":76,"updateProfile":36,"total":47427} (ms)
```

**Висновок:** `buildShoppingList` = **0.2 мс** (чиста синхронна функція, без AI/мережі).
Увесь список покупок = ~76 мс. Вузьке місце — **виклик OpenAI = 47с (99.2%)**.

## Ключовий закон

**Латентність LLM ≈ (кількість вихідних токенів) ÷ (пропускна здатність моделі).**
Тому два важелі: (1) менше токенів на запит, (2) швидша модель, (3) **паралелити**,
щоб wall-clock = найдовший окремий виклик, а не сума.

## Виміряні дані по моделях

Пер-спробний лог: `[generateOneDay] model=… day=… ms=… completionTokens=… (… tok/s)`.

| Модель | Режим | tok/s | токенів/день | Час | Якість рецептів |
|---|---|---|---|---|---|
| gpt-4o | 7 днів, 1 виклик послідовно | ~179* | ~1200* | **47с** | добра |
| gpt-4.1-mini | 4 дні, 1 виклик послідовно | ~95 | ~1150 | 47с | добра |
| gpt-5-mini | паралель по днях | ~80-90 | ~1800 | ~20с/день, викид 44с (tail latency) | добра |
| gpt-5-nano | паралель, прості рецепти | **150-170** | ~1300 | **~10с** перший показ | простіші |
| gpt-5.4-nano | паралель, детальні рецепти | ~135 | ~2100 | ~17с | простіші |
| **gpt-4.1-mini** | **паралель, детальні рецепти** | ~115 | ~1650 | **~15с** | **на висоті ✓** |

\* для gpt-4o tok/s/токени — оцінка (заміри tok/s додані пізніше).

**Підсумок:** gpt-4.1-mini з паралеллю + детальними рецептами — найкращий баланс:
~15с перший показ і **найкраща якість рецептів** (пише концентрованіше — менше
токенів за кращий зміст, тому навіть швидша за nano попри нижчий tok/s).
gpt-5-nano — найшвидша (~10с), але рецепти слабші.

## Що було реалізовано (лежить у stash)

1. **Паралель по днях** — `generateMenuDays()` фаєрить окремий OpenAI-виклик на
   КОЖЕН день через `Promise.all` (`generateOneDay`). Wall-clock ≈ один день, не сума.
   Кожен день ретраїться окремо (3 спроби). Різноманітність між паралельними днями —
   через `DAY_VARIETY_HINTS` (стильові підказки, бо виклики не бачать один одного).

2. **Розбивка тижня на половини + scheduling**:
   - H1 = Пн-Ср `[0,1,2]`, H2 = Чт-Нд `[3,4,5,6]` (індекси 0=Пн…6=Нд).
   - `generate` робить **«сьогодні → кінець поточної половини»** (минулі дні
     пропускаються повністю — і інша половина, і минулі дні поточної).
   - Якщо сьогодні в H1 — майбутня H2 догенеровується **у фоні** через новий
     `POST /api/menu/generate-rest` (квоту генерацій НЕ витрачає; `$push` нових днів
     + атомарний claim на `pendingDayIndices.0 $exists`, щоб не дублювати/не затирати).
   - Клієнт (`menu/page.tsx`) — ефект, що бачить `menu.pendingDayIndices` і тихо
     докочує решту (самовідновлення, якщо вкладку закрили рано); індикатор «🌀 Доганяємо…».

3. **Частковий тиждень**: `WeeklyMenu.days` може мати <7 днів. Тип отримав
   `pendingDayIndices?: number[]`. `buildShoppingList` рахує індекс дня з `day.date`
   (Mon-based `(getDay()+6)%7`), а НЕ з позиції в масиві — інакше для Чт-Нд індекси з'їхали б.
   `dayLabel`/`date` беруться авторитетно з індексу (не з відповіді LLM → і апостроф у
   «П'ятниця» більше не ламає нічого).

4. **Адаптивний `callOpenAI`** під різні родини моделей:
   - reasoning-моделі (`/^(gpt-5|o\d)/`) → `max_completion_tokens` (НЕ `max_tokens`),
     БЕЗ кастомної `temperature`, `reasoning_effort`.
   - класичні (gpt-4o, gpt-4.1-mini) → `max_tokens` + `temperature: 0.7`.

5. **Детальний промт рецептів** — у персоні секція RECIPE: `description` = ПОКРОКОВА
   технологія приготування (нарізка → послідовність → спосіб → вогонь/температура →
   час кроку → коли що додавати → готовність), 4-7 пронумерованих кроків через `\n`,
   конкретні час/температура, + живий приклад. Дубльовано в українському user-промті.
   Рендериться `RecipeTab` через `whitespace-pre-line`. Ліміт токенів/день підняли до 5000.

6. **Ліміт генерацій 3 → 6** на тиждень + показ повідомлення про ліміт у стані
   `has-menu` (раніше повідомлення губилось — банер не рендерився поза `no-menu`).

## ⚠️ Gotchas reasoning-моделей (важливо!)

- **`max_completion_tokens`**, а не `max_tokens`. Інакше 400.
- **Без кастомної `temperature`** (приймають лише дефолт). Інакше 400.
- **`reasoning_effort` — значення РІЗНІ по моделях** (підтверджено помилками API):
  - `gpt-5-nano` приймає `'minimal'`.
  - `gpt-5.4-nano` приймає `'none' | 'low' | 'medium' | 'high' | 'xhigh'`, але **НЕ `'minimal'`**.
    Помилка: `400 Unsupported value: 'reasoning_effort' does not support 'minimal' with this model`.
  - `'none'` = взагалі без reasoning = найшвидше (ідеально для структурованого JSON).
- SDK `openai@^6.22.0` — підтримує всі ці параметри.

## Вердикт

Обрано **`gpt-4.1-mini`** (баланс якість/швидкість). Наразі в код влита **лише
зміна моделі**; уся решта (паралель, half-split, generate-rest, детальні рецепти,
ліміт 6) — у stash.

## Як відновити повну реалізацію

```bash
git stash list          # знайти stash з міткою "menu-gen perf experiments"
git stash pop           # повернути всі зміни (вирішити конфлікт на
                        # generateMenuWithAI.ts — рядок моделі)
```
**Увага:** виправлень списку покупок (баги 1/2/3: `quantityByDay`, ре-синк при свопі,
бекфіл) у stash НЕМАЄ — їх **закомічено окремо в `f22c5e4`** і вони живі в коді. Stash
несе лише роботу з генерації меню + два твіки для часткового тижня (`pendingDayIndices`,
індекс дня з дати).

## Рекомендований план на наступний діалог

1. Влити паралель по днях + skip-past-days + детальний промт рецептів (з stash).
2. Лишити `gpt-4.1-mini` (або перетестувати `gpt-5-nano` для швидкості, якщо
   якість рецептів стане прийнятною).
3. Прибрати тимчасові логи `[menu/generate timing]` та `[generateOneDay]`.
4. Винести ліміт генерацій у `.env` (напр. `MENU_GEN_LIMIT`), щоб не правити код.
5. Окремо — канонізація одиниць/назв у списку покупок (баг №2, друга частина).
