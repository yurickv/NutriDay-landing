## 🔴 Критичні доповнення (must-have)

### 1. Трекер прогресу ваги

Для цільової аудиторії (жінки для схуднення) це **ключова мотиваційна функція**, якої зараз немає.

**Колекція `weight_logs`:**

```js
{
  userEmail: string,
  date: Date,
  weight: number,         // кг
  note: string | null,    // "після свят", "тренування"
  createdAt: Date
}
```

**Що додати:**

- Сторінка/секція з графіком ваги (міні-чарт на головній + повний в профілі)
- Нагадування зважуватись 1 раз на тиждень (не частіше — щоб не провокувати нездорову фіксацію)
- Показувати динаміку: "−1.2 кг за 3 тижні" — це найпотужніший мотиватор
- При зміні ваги — автоматичний перерахунок TDEE та goalCalories
- **Важливо**: не показувати щоденні коливання, орієнтувати на тижневий/місячний тренд

**UI**: Компонент `WeightProgressCard` на головній + `WeightLogPage` в профілі

---

### 2. Відмітка прийомів їжі (а не цілого дня)

Зараз є тільки `isCompleted` на рівні дня. Це занадто грубо — користувачка хоче бачити прогрес протягом дня.

**Зміна в моделі `AIMeal`:**

```js
{
  // ...існуючі поля...
  isConsumed: boolean,       // чи з'їла
  consumedAt: Date | null
}
```

**Що це дає:**

- Прогрес-бар дня: "2 з 4 прийомів ✓"
- Калорії поділяються на з'їдені / заплановані
- `isCompleted` дня = автоматично true коли всі meals consumed
- Стрік рахується якщо з'їдено хоча б 3 з 4 прийомів (а не обов'язково 100%)

---

### 3. Фільтрація списку покупок по днях

Зараз shopping list — це весь тиждень одним списком. Але мало хто купує на цілий тиждень.

**Додати:**

- Фільтр: "На 2-3 дні" / "На весь тиждень"
- Grouped view: "Потрібно для Пн-Ср" / "Потрібно для Чт-Нд"
- Враховувати multi-day prep: якщо борщ готується в понеділок на 3 дні — інгредієнти йдуть у першу групу

**Зміна в `ShoppingListItem`:**

```js
{
  // ...існуючі поля...
  forDays: string[]    // ["Понеділок", "Вівторок", "Середа"]
}
```

---

### 4. Offline-режим для списку покупок

Це **критично** — користувачка в магазині може не мати стабільного інтернету.

**Реалізація:**

- Service Worker кешує останній shopping list
- Відмітки "куплено" зберігаються в IndexedDB
- При відновленні з'єднання — синхронізація з сервером (conflict resolution: last-write-wins з таймстемпами)
- Візуальний індикатор offline-режиму

---

### 5. Порції та адаптація кількості

Зараз `servingSize` є, але немає механізму зміни кількості порцій.

**Додати:**

- Перемикач кількості порцій в рецепті (1 / 2 / 3 порції)
- При зміні — пропорційний перерахунок інгредієнтів
- Це пов'язано з multi-day prep: "Борщ на 3 дні = 6 порцій"
- Оновлення shopping list при зміні порцій

## 🟡 Важливі доповнення (high priority)

### 6. Підсумок макронутрієнтів (не тільки калорії)

Зараз CalorieProgressBar показує тільки ккал. Для схуднення критично важливий баланс БЖВ.

**Додати:**

- Денний прогрес: білки / жири / вуглеводи (г та %)
- Цільові значення БЖВ (рахувати від goalCalories: ~30% білки, ~25% жири, ~45% вуглеводи — або генерувати AI)
- Компонент `MacroProgressBar` під CalorieProgressBar
- В MealCard показувати міні-БЖВ іконки (Б: 25г / Ж: 8г / В: 40г)

### 7. Механізм зворотного зв'язку на страви

Щоб AI з часом генерувало кращі меню:

**Додати в `AIMeal`:**

```js
{
  rating: 1 | 2 | 3 | null,    // 😍 / 😐 / 👎
  ratedAt: Date | null
}
```

**Використання:**

- Після відмітки "з'їла" — показати швидкий рейтинг (3 емодзі, один тап)
- Зберігати та передавати в промпт при наступній генерації:
  ```
  Страви з високим рейтингом (повторити подібні): [...]
  Страви з низьким рейтингом (не повторювати): [...]
  ```
- Це значно покращить персоналізацію з часом

### 8. "Час приготування" у плані дня

Якщо обід готується 45 хв — користувачка має знати це заздалегідь.

**Додати на UI:**

- "Швидкий день ⚡" бейдж якщо всі страви ≤ 20 хв
- "Підготовчий день 🍳" якщо є складна страва
- Сортувати snacks по часу (найшвидші зверху)

### 13. "Швидка заміна" без AI

Зараз swap завжди викликає OpenAI (дорого + повільно). Додати:

- Список з 3-5 швидких альтернатив, згенерованих разом із основним меню
- AI генерує `alternatives: AIMeal[]` для кожної страви одразу
- Swap через AI — тільки як fallback "Інші варіанти..."

**Зміна в промпті:**

```
Для кожної страви також запропонуй 2 швидкі альтернативи з тією ж калорійністю (±10%).
```

**Зміна в `AIMeal`:**

```js
{
  // ...існуючі поля...
  quickAlternatives: AIMeal[]  // 2-3 передгенеровані заміни
}
```

> **Увага**: це збільшить розмір відповіді OpenAI та вартість. Оцінити token usage і вирішити чи виправдано.

## 🛡️ Безпека та edge cases

### 16. Захист від нездорових патернів

Оскільки ЦА — жінки для схуднення, є ризик розладів харчування:

- **Мінімальна калорійність**: не дозволяти goalCalories < 1200 ккал для жінок
- **Попередження**: якщо BMI < 18.5 — показувати м'яке повідомлення і рекомендувати консультацію спеціаліста
- **Без негативного фреймінгу**: "Ви перевищили калорії" → "Сьогодні вийшло трохи більше — це нормально, завтра новий день!"
- **Стріки**: не карати за пропуск, а позитивно підкріплювати продовження ("Повернулась! Це вже перемога 🎉")
- **Не показувати**: різку зміну ваги як "провал"

---

### 17. Rate limiting та error handling для AI

Додати до плану:

- **Rate limit**: максимум 3 генерації на тиждень на користувача
- **Retry логіка**: якщо OpenAI повертає помилку — 3 ретраї з exponential backoff
- **Fallback**: якщо AI недоступний — показати останнє активне меню
- **Валідація**: перевіряти що AI не вигадав неіснуючі продукти (базова sanity check)
- **Token budget**: встановити `max_tokens` щоб не перевищувати бюджет

---

### 18. Конфлікт multi-day prep та swap

**Edge case**: Користувачка хоче замінити обід у вівторок, але це борщ який готувався в понеділок (multi-day).

**Рішення:**

- При swap — попередження: "Цю страву приготовлено в понеділок на 3 дні. Замінити тільки вівторок чи всі дні?"
- Варіанти: замінити один день / замінити всі пов'язані дні
- Перерахунок shopping list після swap
